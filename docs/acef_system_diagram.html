How ACEF Works — End to End

An AI compliance evidence bundle starts as structured data, gets signed and sealed into a deterministic archive, then validated against regulation templates to produce an assessment verdict.

1
Build
Package
🏗

Initialize Package

Create a new evidence package with producer metadata, retention policy, and optional link to a prior bundle.

Package(producer=...)
🤖

Declare Subjects

Register the AI system or model being documented — its type, risk class, lifecycle phase, and modalities.

pkg.add_subject(...)
🔗

Build Entity Graph

Add components (model, guardrail, retriever), datasets (training, eval), actors (provider, deployer, auditor), and relationships between them.

pkg.add_component(...)
📋

Attach Profiles

Declare which regulations apply — EU AI Act, NIST RMF, China CAC, etc. Each profile lists the provisions to evaluate against.

pkg.add_profile(...)
2
Record
Evidence
📝

Add Evidence Records

Each record has a type (risk_register, dataset_card, etc.), a payload with type-specific data, provisions addressed, entity refs, and trust level.

pkg.record("risk_register", ...)
📎

Attach Artifacts

Binary files — evaluation reports (PDF), model cards (Markdown), audit logs. Each attachment gets a SHA-256 hash and MIME type.

pkg.add_attachment(...)
🔐

Set Confidentiality

Each record carries a confidentiality level: public, redacted, hash-committed, regulator-only, or under-NDA.

3
Seal &
Sign
🔒

Compute Integrity

SHA-256 hash every file via RFC 8785 canonicalization. Build a Merkle tree with odd-leaf promotion. Write content-hashes.json and merkle-tree.json.

✍️

Sign with JWS

Create a detached JWS (RFC 7515) over content-hashes.json using RS256 or ES256. Supports multiple signers (provider + auditor).

pkg.sign(key_path)
🔏

Optional: Redact

Create a redacted copy — sensitive payloads replaced with SHA-256 hash commitments. Integrity remains verifiable.

acef.redact(pkg, spec, path)
4
Export
Bundle
📁

Directory Bundle

Content-addressable directory: acef-manifest.json, records/ (JSONL), artifacts/, hashes/, signatures/.

pkg.export("./out.acef/")
📦

Archive Bundle

Deterministic .acef.tar.gz — gzip level 6, mtime=0, OS=0xFF, owner 0/0, perms 0644. Byte-identical across implementations.

pkg.export("out.acef.tar.gz")
5
Validate
Evidence
✅

4-Phase Pipeline

Schema → Integrity → References → Rules. Each phase collects all errors before proceeding. 60 error codes (ACEF-001 to ACEF-060).

acef.validate(path, profiles)
⚙️

DSL Rule Engine

10 operators evaluate evidence against regulation templates: has_record_type, field_value, evidence_freshness, exists_where, etc.

📊

Provision Rollup

7-step precedence algorithm collapses per-rule results into a single provision outcome: SATISFIED, NOT_SATISFIED, PARTIALLY_SATISFIED, GAP_ACKNOWLEDGED, SKIPPED, or NOT_ASSESSED.

6
Produce
Assessment
📜

Assessment Bundle

A separate .acef-assessment.json file: per-rule results, per-provision summaries, per-subject breakdowns, structural errors, template digests. Optionally signed.

📑

Human Report

Render assessment as Markdown or console output — compliance officers and auditors get a readable summary with evidence links.

acef.render(assessment)
🔁

Chain & Iterate

Link the next evidence bundle to this one via prior_package_ref. Continuous compliance: each snapshot adds to the chain.

acef.chain(prior_path)

✅ Inside hash domain

acef-manifest.json

The envelope — package-level metadata

This is the root document of every ACEF bundle. It contains everything about the package except the evidence records themselves (which live in records/). All JSON is RFC 8785-canonicalized for deterministic hashing.

Metadata
package_id, timestamp, producer, retention_policy, prior_package_ref
Versioning
core_version, profiles_version (semver, independently versioned)
Subjects
AI systems and models (type, risk class, lifecycle phase, modalities)
Entities
Components, datasets, actors, relationships (W3C PROV graph)
Profiles
Regulation profiles with applicable provision lists
Audit Trail
Package-level audit history entries
✅ Inside hash domain

records/

Evidence in deterministic JSONL format

One file per record type. Each line is an RFC 8785-canonicalized JSON object representing a single evidence record wrapped in the common envelope (record_id, record_type, provisions_addressed, timestamp, payload, entity_refs, etc.).

Records are sorted deterministically: primary by timestamp ascending (ISO 8601), secondary by record_id ascending. This ensures any conformant implementation produces identical files.

✅ Inside hash domain

*.jsonl Record Files

One record per line, canonicalized

Each line is a complete RecordEnvelope serialized as RFC 8785-canonical JSON followed by a single newline (0x0A). No empty lines, no BOM. The file is hashed line-by-line — each line is independently canonicalized before contributing to the file's SHA-256 hash.

Envelope Fields
record_id, record_type, timestamp, provisions_addressed, entity_refs, obligation_role, confidentiality, trust_level
Payload
Type-specific evidence data validated against per-type JSON Schema
✅ Inside hash domain

Sharded Record Directories

Automatic splitting for large record sets

When a record type exceeds 100,000 records or 256 MB, it moves from a single .jsonl file to a sharded directory. Shard files use zero-padded 4-digit numbering: event_log.0001.jsonl, event_log.0002.jsonl, etc.

Shard boundaries are deterministic — every conformant implementation splits at the same points, producing identical shard files.

✅ Inside hash domain

artifacts/

Binary attachments referenced by evidence records

PDFs, images, Markdown documents, logs — any file that backs an evidence claim. Each artifact is referenced from a record's attachments array with a path, MIME type, and advisory SHA-256 hash. Binary files are hashed in 64 KB streaming chunks (not canonicalized).

Path traversal is blocked at multiple layers: add_attachment(), export, and load all validate paths independently.

🚫 Outside hash domain

hashes/

Integrity verification layer

This directory sits outside the hash domain — it contains the hashes themselves, so it can't be part of what's hashed (that would create a circular dependency). It holds two files: content-hashes.json and merkle-tree.json.

🚫 Outside hash domain

content-hashes.json

SHA-256 hash index of every file in hash domain

A flat JSON object mapping every file path (manifest, records, artifacts) to its SHA-256 hex digest. Verifiers recompute each hash and compare. This is the document that JWS signatures are computed over — signing content-hashes.json effectively signs the entire bundle.

🚫 Outside hash domain

merkle-tree.json

Merkle tree over content hashes

Contains a sorted leaf array (path + hash pairs) and a computed root hash. Leaf hashes are SHA-256(path_bytes || 0x00 || hash_hex_bytes). Inner nodes use SHA-256(left_raw_32_bytes || right_raw_32_bytes). Odd leaves are promoted, not duplicated.

The root hash provides a single value that represents the entire bundle's integrity — change any file and the root changes.

🚫 Outside hash domain

signatures/

Detached JWS signatures

Multiple parties can sign the same bundle independently — a provider signs first, then an auditor or notified body adds their signature. Each .jws file is a detached JWS (RFC 7515) computed over content-hashes.json.

🚫 Outside hash domain

*.jws Signature Files

RFC 7515 detached signatures — RS256 or ES256 only

Format: header..signature (empty payload section). The JWS header carries alg (RS256 or ES256), optional kid (key identifier), optional x5c (certificate chain), and auto-derived jwk (public key). All other algorithms are rejected with ACEF-013.

Payload
content-hashes.json (canonical bytes)
Algorithms
RS256, ES256 (only these two)

Meta

Validation Pipeline

Four sequential phases. Each collects all errors before stopping. Click a phase to see what it checks.

Phase 1

Schema

JSON Schema validation for manifest, envelopes, and payloads

Phase 2

Integrity

SHA-256 hashes, Merkle tree root, JWS signature verification

Phase 3

References

Entity URN resolution, file paths, duplicates, attachment hashes

Phase 4

Rules

DSL operator evaluation, provision rollup, per-subject assessment

Phase 1: Schema Validation

Verifies the structural correctness of every JSON document in the bundle against the ACEF JSON Schema registry.

  • ✔Validate acef-manifest.json against manifest.schema.json
  • ✔Validate each record's common envelope against record-envelope.schema.json
  • ✔Validate each record's payload against its type-specific schema (e.g., risk_register.schema.json)
  • ✔Check module version compatibility (core_version, profiles_version)
  • ✔Verify record_type is one of the 16 v1 types or a valid x- extension
ACEF-001ACEF-002ACEF-003ACEF-004

Phase 2: Integrity Verification

Verifies that no file has been tampered with since the bundle was sealed.

  • ✔Recompute SHA-256 of every file in the hash domain (manifest, records, artifacts)
  • ✔Compare computed hashes against content-hashes.json — mismatch = ACEF-010
  • ✔Verify content-hashes.json is complete (no missing files, no extra entries) — ACEF-014
  • ✔Recompute Merkle tree from content hashes; verify root matches merkle-tree.json — ACEF-011
  • ✔If signatures present: verify each JWS against its public key; check algorithm is RS256 or ES256 — ACEF-012, ACEF-013
ACEF-010ACEF-011ACEF-012ACEF-013ACEF-014

Phase 3: Reference Checking

Verifies all internal references are consistent — no dangling pointers, no duplicates, no missing files.

  • ✔Resolve every entity_ref URN in every record — dangling refs = ACEF-020
  • ✔Check for duplicate URNs across entities — ACEF-021
  • ✔Verify record_files paths in manifest point to existing files — ACEF-022
  • ✔Verify attachment paths are under artifacts/ and exist — ACEF-023
  • ✔Verify record counts match manifest declarations — ACEF-025
  • ✔Check for duplicate record_id values — ACEF-026
  • ✔Verify attachment advisory hashes (warning if mismatch) — ACEF-027
ACEF-020ACEF-021ACEF-022ACEF-023ACEF-025ACEF-026ACEF-027

Phase 4: Rule Evaluation

Evaluates evidence against regulation template rules using the DSL engine, then rolls up per-provision outcomes.

  • ✔Load regulation templates for each declared profile
  • ✔For each subject × provision × rule: evaluate DSL operator with record set
  • ✔10 operators: has_record_type, field_present, field_value, evidence_freshness, exists_where, attachment_kind_exists, all_records_match, count_satisfies, redaction_compliant, scope_filter
  • ✔Apply empty-set semantics: existential operators FAIL on zero records, universal operators PASS vacuously
  • ✔Run 7-step provision rollup: errors → future → gap → fail → warning → info → all-pass
  • ✔Produce per-subject provision summaries and per-rule evidence links
